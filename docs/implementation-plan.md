# План реализации `dlisio` на C# (.NET Framework 4.8)

## Короткий вывод

Реализация **возможна**. Основной риск не в синтаксисе C#, а в
корректной интерпретации RP66/DLIS (индексы, объектная модель, edge-cases)
и в достижении сопоставимой производительности на больших файлах.

Сохранение C++ вставок допустимо и может заметно снизить риски по скорости.

---

## Что нужно реализовать

1. **Бинарный парсер DLIS/RP66**
   - чтение физических/логических записей;
   - обработка framing, offsets, segment chaining;
   - строгая валидация структуры.
2. **Объектная модель DLIS**
   - channels, frames, parameters, tool/string metadata;
   - кросс-ссылки, lazy loading, разрешение индексов.
3. **Декодирование типов данных**
   - числовые типы, строки, time/date, массивы;
   - vendor-specific и нестандартные значения.
4. **Публичный API**
   - безопасный, предсказуемый API уровня библиотеки;
   - исключения, диагностика, частичное чтение.
5. **Производительность**
   - потоковое чтение, минимизация аллокаций;
   - оптимизация горячих участков (в т.ч. C++).
6. **Совместимость и верификация**
   - сравнение поведения с оригинальным `dlisio`;
   - набор эталонных файлов/снимков результатов.

---

## Предлагаемая архитектура

## Вариант A: Чистый C#

- Ядро парсера и модель в C#.
- Unsafe/Span-like подходы ограниченно (учитывая net48).
- Плюсы: проще CI и поддержка.
- Минусы: больше работы по ручной оптимизации.

## Вариант B: C# + C++/CLI

- Сложные/горячие декодеры реализуются в C++.
- C#-слой предоставляет основной API.
- Плюсы: удобный вызов C++ из .NET.
- Минусы: сильная зависимость от toolchain Visual Studio/MSBuild Windows.

## Вариант C: C# + native C++ + P/Invoke

- Нативная DLL для производительных модулей.
- C# вызывает через стабильный C ABI.
- Плюсы: гибко, при желании можно переиспользовать C++ код.
- Минусы: marshalling и контракт ABI надо проектировать аккуратно.

**Оценка трудоёмкости:** высокая.

## Насколько это долго

Без оценки по календарным дням: проект имеет **высокую суммарную
трудоёмкость** и много инженерных рисков в доменной части формата DLIS.

Если мерить относительными единицами:

- MVP (чтение + базовая модель + базовые типы): **крупная задача**;
- Совместимость уровня production с оригиналом: **очень крупная задача**;
- Производительность, близкая к C++/оригиналу: обычно требует
  отдельного оптимизационного цикла.

---

## Главные риски

1. Неполная совместимость на редких/грязных полевых файлах.
2. Сложность воспроизведения точного поведения `dlisio`.
3. Усложнение сборки при C++/CLI для .NET Framework 4.8.
4. Рост стоимости поддержки при смешанном C#/C++ стеке.

---

## Рекомендация старта

1. Начать с **чистого C# ядра** и корректности формата.
2. Заложить `Interop` слой как опциональный модуль заранее.
3. Добавлять C++ только по результатам профилирования.
4. С самого начала строить regression suite на реальных DLIS-файлах.
